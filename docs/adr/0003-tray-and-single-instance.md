# 0003 通知領域と単一インスタンスを標準APIで構成する

- 状態：Accepted
- 日付：2026-07-15

## 文脈

MVPは通知領域から主要操作へ到達でき、ウィンドウを閉じても手動スリープ防止を維持する必要があります。
二回目の起動は電源状態を変更せず、既存ウィンドウを表示して終了する必要があります。
外部UIライブラリを採用する場合は、.NET 10対応、更新頻度、ライセンス、ネイティブリソースの解放を追加で管理する必要があります。

## 判断

通知領域は[`Shell_NotifyIconW`](https://learn.microsoft.com/en-us/windows/win32/api/shellapi/nf-shellapi-shell_notifyiconw)を直接呼び出します。
MVPはウィンドウハンドルと数値IDでアイコンを識別し、`NOTIFYICON_VERSION_4`のコールバック形式を使用します。
左クリックとキーボード選択はウィンドウを表示し、右クリックはWPFのコンテキストメニューを開きます。
`TaskbarCreated`を受けた場合はアイコンを再登録し、終了時は`NIM_DELETE`で削除します。
共有のシステムアイコンは破棄せず、コンテキストメニューを閉じた後は`NIM_SETFOCUS`を送ります。

単一インスタンスは、現在のユーザーとセッションへ限定した名前付きMutexを生存標識として使用します。
二回目のプロセスは同じ範囲のAutoResetイベントを通知し、既存プロセスがDispatcher上でウィンドウを表示します。

起動時は通常のWPFウィンドウを表示します。
ウィンドウの閉じる操作と「隠す」は通知領域へ格納し、画面または通知領域の「終了」だけがプロセスを終了します。

## 結果

本番NuGet依存を増やさず、通知領域アイコンと単一インスタンスの所有権をアプリの終了処理へ集約できます。
二重起動で既存ウィンドウが再表示され、二回目のプロセスが終了することを実機で確認しました。
Windows 10、複数DPI、Explorer再起動後の見た目と操作は実機確認が残っています。

数値IDは実行ファイルの移動に影響されませんが、Explorerを越えて永続するアイコン識別子にはなりません。
将来MSIXや固定インストール先を採用する場合は、GUID識別へ移行するかを再評価します。

## 比較した案

外部の通知領域ライブラリは実装量を減らせますが、MVPに必要な操作が小さく、依存更新とライセンス管理の方が大きいため採用しませんでした。
Mutexだけで二回目を終了する案は、既存ウィンドウを再表示できないため採用しませんでした。
GUIDによるアイコン識別は配布経路が確定していない段階では移動可能なzip配布と相性が悪いため、MVPでは採用しませんでした。
