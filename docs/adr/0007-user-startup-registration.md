# 0007 現在ユーザーのスタートアップ登録

- 状態：Accepted
- 日付：2026-07-20

## 文脈

DisplayLightは通知領域から操作する常駐アプリであり、Windowsへサインインした後に利用者が毎回手動で起動すると、スリープ防止を使う前の準備が増えます。
自動起動の有無はDisplayLight独自の設定画面ではなく、Windowsのタスクマネージャーから切り替えられる必要があります。
BatteryMonitorはタスクスケジューラを使用しますが、今回の管理場所はタスクマネージャーの「スタートアップ アプリ」です。

## 判断

インストール版は、現在ユーザーの`HKCU\Software\Microsoft\Windows\CurrentVersion\Run`へ`DisplayLight`という値を登録します。
値にはVelopackのルートにあるランチャーの絶対パスと`--startup`を保存します。
Windowsは`Run`登録をタスクマネージャーの「スタートアップ アプリ」へ表示します。

登録はインストール直後、更新直後、通常のインストール版起動時に不足している場合だけ作成します。
既存値が期待値と同じ場合は書き直さず、タスクマネージャーで利用者が選んだ無効状態を維持します。
アンインストール時はVelopackのフックから登録を削除します。
Portable版と開発時実行は登録しません。

`--startup`による起動では通知領域アイコンと内部状態だけを初期化し、フライアウトを表示しません。
同じ時点ですでにプロセスが動作している場合も、スタートアップ起動は既存フライアウトを前面化しません。
通常の明示起動は従来どおりフライアウトを表示します。

## 結果

Windowsサインイン後はDisplayLightが画面を占有せず通知領域へ常駐します。
利用者はタスクマネージャーから自動起動を無効化し、必要になった場合は同じ場所から再度有効化できます。
登録はユーザー単位であり、管理者権限を要求しません。
コード署名を導入するまでは、タスクマネージャーに発行元を確認できない表示が出る可能性があります。

## 比較した案

タスクスケジューラはBatteryMonitorと実装を共有しやすい一方、利用者が求める管理場所と一致しないため採用しませんでした。
Startup Taskはパッケージマニフェストと相性がよい方式ですが、現在のVelopackによる非MSIX配布へ追加のパッケージ境界を持ち込むため採用しませんでした。
スタートアップフォルダーのショートカットはタスクマネージャーへ表示できますが、更新に追随するランチャーをレジストリ値で直接指定する方が構成を少なくできます。
