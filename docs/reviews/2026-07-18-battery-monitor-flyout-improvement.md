# BatteryMonitorのフライアウト改善提案

- 作成日：2026年7月18日
- 対象：通知領域から開くWPFフライアウト
- 検証元：DisplayLightのWindows 11向けフライアウト実装

## 文書の位置づけ

この文書は、BatteryMonitorの通知領域フライアウトをWindows 11のコントロールセンターに近い動作へ改善するための実装メモです。
DisplayLightで実機確認した現象と修正結果をもとにしています。

BatteryMonitorの現行コードは、この文書の作成時点では再調査していません。
したがって、イベント名やクラス名はWPFの一般的な構成を示す例であり、BatteryMonitorへ組み込む際には既存のウィンドウ管理方式へ読み替える必要があります。

## 改善対象

通知アイコンから開くフライアウトでは、次の動作を一つの表示ライフサイクルとして扱います。

1. 通知アイコンの位置を基準に最終表示位置を決める。
2. フライアウト面をタスクバー側へ完全に隠した状態でウィンドウを表示する。
3. 面全体をタスクバーの裏から最終位置まで移動する。
4. 面の移動完了後に内容をフェードインする。
5. 通知アイコンの再クリック、領域外クリック、Escapeで閉じる。
6. 閉じる際は面をタスクバー側へ戻し、移動完了後にウィンドウを非表示にする。

領域外クリックでは、クリック先のウィンドウがアクティブになっても閉じるモーションを手前に表示します。
クリック直後に背後の画面を一瞬見せないことも、この動作に含めます。

## 領域外クリックでアニメーションが見えない原因

WPFの`Deactivated`は、別のウィンドウがアクティブになった結果として発生します。
このイベントから閉じるアニメーションを開始しても、クリック先のウィンドウがすでにZオーダーの手前へ移っている場合があります。

その状態では、フライアウトは背後でアニメーションしています。
利用者にはクリックした瞬間に消えたように見えるため、`Hide()`の呼び出しが早い問題と誤認しやすくなります。

`Deactivated`内で`Topmost = true`へ変更するだけでも、閉じるモーション自体は手前へ戻せます。
しかし、クリック先が手前へ移った後にフライアウトを昇格させる順序になるため、切り替えの間に背後の画面が一瞬見えることがあります。

## 表示期間を通したZオーダー維持

フライアウトは、表示開始前から非表示完了までTopmostバンドに置きます。
これにより、領域外クリック時の「通常バンドからTopmostバンドへの昇格」が不要になり、閉じる面を連続して表示できます。

```csharp
private async Task ShowFlyoutAsync()
{
    Topmost = true;

    PrepareHiddenSurface();
    Show();
    Activate();

    await AnimateSurfaceAsync(isOpening: true);
    await FadeInContentAsync();
}

private async Task HideFlyoutAsync()
{
    if (!IsVisible || isClosing)
    {
        return;
    }

    isClosing = true;
    FlyoutContent.IsHitTestVisible = false;

    try
    {
        await AnimateSurfaceAsync(isOpening: false);
        Hide();
    }
    finally
    {
        Topmost = false;
        FlyoutContent.IsHitTestVisible = true;
        isClosing = false;
    }
}
```

このTopmost指定は、アプリケーションを常時最前面に固定する設定ではありません。
フライアウトが見えている期間だけ使用し、`Hide()`の直後と例外経路で必ず解除します。

## 非アクティブ化の判定

領域外クリック相当の判定には`LostFocus`ではなく`Window.Deactivated`を使用します。
`LostFocus`はフライアウト内の子コントロール間でも発生するため、スライダーやメニューの操作を閉じる動作と誤判定する可能性があります。

```csharp
private async void Window_Deactivated(object sender, EventArgs e)
{
    await Task.Delay(120);

    if (!isAuxiliaryMenuOpen && !IsActive && IsVisible)
    {
        await HideFlyoutAsync();
    }
}
```

短い待機はアニメーションのためではなく、コンテキストメニューなど一時的な非アクティブ化を判定するために置きます。
待機後に`IsActive`と補助UIの状態を再確認し、閉じる必要がなくなった場合は何もしません。

補助メニューを別のWPF `Popup`や`ContextMenu`で表示する場合は、開閉状態を明示的に追跡します。
メニューが開いたことによる非アクティブ化では親フライアウトを閉じず、メニューが閉じた後に親の状態を再評価します。

## ウィンドウではなく面を動かす

開閉アニメーションでは、トップレベルHWNDをフレームごとに移動しません。
HWNDを最終位置へ固定し、その内側で背景、外周、影、内容を含む一つの面を移動します。

`SetWindowPos`によるHWND移動とWPFの描画面は、同じ描画トランザクションで更新されません。
実機では両者の更新差によって、Desktop Acrylicの灰色面や未描画領域が約数フレーム露出しました。

推奨構成は次のとおりです。

```text
固定したトップレベルWindow
└─ クリップ領域
   └─ FlyoutSurface
      ├─ 背景
      ├─ 外周と影
      └─ FlyoutContent
```

`FlyoutSurface`へ`TranslateTransform`を設定し、タスクバーに垂直な方向へ移動します。
位置更新は固定間隔の`DispatcherTimer`ではなく、`CompositionTarget.Rendering`へ同期させます。

DisplayLightでは、開く動作に約250ミリ秒のcubic ease-out、閉じる動作に約170ミリ秒のcubic ease-inを使用しています。
数値の完全な移植よりも、完全に隠れた位置から始まることと、背景を含む面全体が同じ進捗で動くことを優先します。

## 内容の表示タイミング

開く途中から文字やボタンを見せると、面と内容が別々に動いて見えます。
外側の面は不透明度100%のまま移動し、最終位置へ到達した後に内容だけを約200ミリ秒でフェードインします。

閉じるときは、入力を直ちに無効化してから面全体を移動します。
これにより、移動中のボタンへクリックが入り、非表示後に処理される競合を防げます。

## 高さ変更

カードの展開や状態メッセージによって高さが変わる場合は、`SizeToContent`による瞬間的な変更を続けません。
内側コンテンツの必要高を測定し、通知アイコン側の辺を固定したまま約180ミリ秒でウィンドウ寸法を補間します。

伸縮中に`ScrollViewer`が一時的なオーバーフローを検出すると、右端へスクロールバーが現れることがあります。
スクロールを提供しないフライアウトでは、`VerticalScrollBarVisibility="Hidden"`として描画経路自体を無効にします。

伸縮中は入力を無効化し、完了、中断、例外のすべての経路から入力状態を復元します。
再計測は進行中の伸縮処理を直接キャンセルせず、現在の処理が終了した後にキューから開始します。

## WPF Popupを使う場合の制約

BatteryMonitorがトップレベル`Window`ではなくWPF `Popup`を使っている場合、この方式をそのまま適用できるとは限りません。
`Popup`は別のネイティブウィンドウとして生成され、アクティブ化、フォーカス、Zオーダーの制御が`Window`と異なるためです。

領域外クリック、キーボード操作、補助メニュー、Topmostの解除を一貫して制御する必要がある場合は、標準タイトルバーとタスクバー表示を持たない小型`Window`への置き換えを先に検証します。
ホバー表示や非アクティブな情報表示を優先する場合は`Popup`を維持する判断もあり、その場合は同一の開閉モデルを前提にしません。

## 競合を防ぐ状態管理

開く、閉じる、再クリック、領域外クリックが重なるため、少なくとも次の状態を明示的に持ちます。

- **表示中**：ウィンドウが可視かどうか。
- **終了処理中**：閉じる処理の多重起動を防ぐ状態。
- **モーションのキャンセル元**：再クリック時に現在位置から反転するための`CancellationTokenSource`。
- **補助UI表示中**：コンテキストメニューによる非アクティブ化を区別する状態。
- **入力可否**：移動または伸縮中の誤操作を防ぐ状態。

キャンセル時に完成位置へ飛ばすと、通知アイコンの連打で不連続な動きになります。
現在の変換値を次のアニメーションの始点として読み取り、そこから反対方向へ補間します。

## 実装順序

1. 現在の表示方式が`Window`か`Popup`かを確認する。
2. 通知アイコンのクリック、領域外クリック、Escapeを一つの開閉処理へ集約する。
3. 表示開始前にTopmostを設定し、非表示完了後に解除する。
4. HWNDを固定し、単一の`FlyoutSurface`を描画フレーム同期で移動する。
5. 開く面の停止後に内容をフェードインする。
6. 再クリックと領域外クリックを含むキャンセル処理を追加する。
7. 必要な場合だけ高さ変更の補間を追加する。

Zオーダーと面移動を同時に変更すると、不具合の原因を判別しにくくなります。
各段階で録画をフレーム単位に分解し、背後の画面、外周、背景面、内容のどれが先に変化したかを確認します。

## 検証項目

- [ ] 通知アイコンのクリックで、完成位置のフレームを先に見せずに開く。
- [ ] 面全体がタスクバーの裏から現れる。
- [ ] 開く面が停止した後に内容が表示される。
- [ ] 通知アイコンの再クリックで閉じるモーションが見える。
- [ ] 領域外クリックでも閉じるモーションがクリック先の手前に見える。
- [ ] 領域外クリックの瞬間に背後の画面が一瞬だけ露出しない。
- [ ] Escapeでも同じ閉じる経路を通る。
- [ ] コンテキストメニュー操作では親フライアウトが不用意に閉じない。
- [ ] 開閉途中の再クリックで面が完成位置へ飛ばない。
- [ ] 非表示後にTopmostが解除され、ほかのウィンドウを遮らない。
- [ ] アプリ終了と例外経路でもTopmostと入力禁止状態が残らない。
- [ ] Windowsのアニメーション設定が無効な場合は即時に開閉する。
- [ ] 100%、150%、200% DPIで通知アイコンとの位置関係が保たれる。
- [ ] 下、上、左、右のタスクバーで隠れる方向が正しい。
- [ ] 異なるDPIの複数モニター間でも配置がずれない。
- [ ] ライト、ダーク、高コントラストで背景と内容を判別できる。

## DisplayLightで確認済みの範囲

DisplayLightでは、固定HWND内の単一面移動、描画フレーム同期、外側クリック後の閉じるモーション、表示期間を通したTopmost維持を実装しています。
Releaseビルドと77件の自動テストは成功しています。

利用者は、領域外クリックでも閉じるアニメーションが見えることを実機で確認しました。
表示期間を通したTopmost維持へ変更した後の「背後の画面が一瞬だけ露出しないこと」は、BatteryMonitorへ移植する前にDisplayLightとBatteryMonitorの両方で目視確認する必要があります。
