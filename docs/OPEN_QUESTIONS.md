# 設計上の判断と未決事項

## 使い方

この文書には、企画仕様書だけでは一意に決められない事項を記録します。
各項目には推奨案を置いていますが、`決定`が`未決`の項目は実装の前提として確定していません。

決定した項目は、決定日と理由を記入し、設計へ影響する場合はADRと`DESIGN.md`も更新します。

## MVPに関する判断

### Q1. 製品名とリポジトリ名

**決定**：2026年7月15日、製品名を`DisplayLight`としました。
ソリューション、プロジェクト、名前空間、AppDataの保存先にも同じ名前を使用します。

### Q2. MVPの完了地点

**不足している情報**：企画書の「MVPに含めるもの」では手動のスリープ防止までですが、「完成条件」には作業完了後の自動復帰と自動スリープのキャンセルが含まれています。

**決定**：2026年7月15日、Codex連携の手前までをMVPとしました。
企画書のPhase 1からPhase 4までをMVPに含め、Phase 5以降はAgent Previewとして扱います。

### Q3. ディスプレイ消灯時間の読み取り方式

**不足している情報**：企画書は変更に`powercfg /change`を指定していますが、現在値を言語非依存で取得する具体的方法がありません。

**決定**：2026年7月15日、読み取りと書き込みをPowrProf APIへ統一しました。
現在の電源プランを取得し、`VIDEOIDLE`のAC値とDC値を秒単位で読み書きします。
変更後は両方を読み直し、対象値が要求値と一致した場合だけ成功と表示します。
実機でAC値とDC値を別々に変更できることを確認し、確認後に元の値へ復旧しました。
判断理由は`docs/adr/0002-windows-power-apis.md`に記録します。

### Q4. 消灯時間変更を終了時に戻すか

**不足している情報**：「通常モード」は消灯時間を変更する設定ツールですが、安全ルールには「すべての電源変更に元へ戻す手段を用意する」とあります。

**決定**：2026年7月15日、消灯時間は永続的なWindows設定変更とし、アプリ終了時に戻さないことにしました。
変更前の値へ戻す専用操作も設けず、利用者はスライダーから別の値へ切り替えます。
一時的なスリープ防止要求だけは、アプリ終了時に解放します。

### Q5. 「無期限」の扱い

**決定**：2026年7月15日、無期限をMVPのプリセットに含めることにしました。
消灯時間は離散値を選ぶスライダーで操作し、一番右を無期限とします。
初期の選択肢は`1分`、`5分`、`10分`、`30分`、`60分`、`無期限`とし、任意の数値を入力するカスタム欄はMVPに含めません。
AC時とバッテリー時は対象を切り替えて、それぞれの値を設定します。

### Q6. スリープ防止API

**不足している情報**：企画書は`SetThreadExecutionState`を候補としていますが、Power Request APIとの比較がありません。

**決定**：2026年7月15日、Power Request APIを採用しました。
`SystemRequired`だけを要求し、ディスプレイ消灯、Away Mode、実行継続は要求しません。
要求ハンドルはプロセスが所有し、利用者の解除、エラー、終了時に解放します。
作成と解除が成功することは実機で確認しましたが、アイドルタイムアウトとModern Standbyの長時間挙動は未確認です。
判断理由は`docs/adr/0002-windows-power-apis.md`に記録します。

### Q7. 低バッテリーの既定値と動作

**決定**：2026年7月15日、独自の低バッテリー下限と自動解除を設けないことにしました。
バッテリー消費を避けたい場合は、AC電源限定の設定を使用します。

### Q8. アプリが起動していないときのHookイベント

**不足している情報**：連携コマンドがWPFアプリを起動するか、イベントを捨てるかが決まっていません。

**推奨案**：Agent Previewでは、連携コマンドがアプリを非表示で起動し、IPCが準備できるまで短時間だけ再試行します。
起動に失敗した場合はゼロ以外で終了しますが、エージェントの本処理自体はHook失敗で止めない設定にします。

**決定**：Agent Previewまで保留します。
MVPはHookを受信しないため、Q8はMVPの設計と実装を妨げません。

## Agent Previewまでに決める項目

### Q9. Codexで「完了」を何から確定するか

**不足している情報**：Codexの`Stop`はターン停止であり、スレッド全体の完了ではありません。
Codex AppとCLIで、作業開始、入力待ち、継続可能、利用者が終えた状態を同じイベント集合から区別できるかも未確認です。

**推奨案**：最初のHook連携では`Working`と`WaitingForInput`だけを自動化し、`Completed`からの自動スリープは実験機能にします。
Hook payload、発火時刻、再開動作を個人情報を除いて記録する診断モードを用意し、実機観測後に完了規則を決めます。

**決定**：未決。

### Q10. 欠落イベントとセッション有効期限

**不足している情報**：Hookが異常終了した場合、`Working`をいつ解除するかが決まっていません。

**推奨案**：アダプターごとにleaseを持ち、Codex Hookだけではheartbeatが得られない場合は、 プロセス検出と利用者通知を組み合わせます。
一律の固定時間で作業完了とは見なさず、期限切れを`Unknown`として要求を解放します。

**決定**：未決。

### Q11. `WaitingForInput`中の扱い

**不足している情報**：企画書では既定で要求を解放しますが、AC接続時でも同じにするか、入力待ち通知を見逃した場合の猶予を置くかが決まっていません。

**推奨案**：既定では即座に要求を解放して通知します。
後から「AC接続時だけ入力待ち中も保持」を詳細設定として追加できますが、初期版には入れません。

**決定**：未決。

### Q12. 複数セッションの表示単位

**不足している情報**：プロジェクトパスをそのまま表示するか、リポジトリ名だけにするか、同じプロジェクトの複数ターンをまとめるかが決まっていません。

**推奨案**：一覧は「連携元、フォルダー名、経過時間、状態」を表示し、完全なパスは明示操作でだけ表示します。
集約キーは連携元とセッションIDにし、作業ディレクトリは識別キーに使用しません。

**決定**：未決。

## MVP実装で確定したUIと配布前の判断

### Q13. 通知領域の実装方式

**不足している情報**：`Shell_NotifyIcon`を直接P/Invokeするか、保守されているライブラリを使うかが決まっていません。

**決定**：2026年7月15日、`Shell_NotifyIconW`の直接呼び出しを採用しました。
MVPはウィンドウハンドルと数値IDでアイコンを識別し、`NOTIFYICON_VERSION_4`を使用します。
Explorerの再起動通知後はアイコンを再登録し、終了時は明示的に削除します。
DPI別表示とExplorer再起動の実機確認は残っています。
判断理由は`docs/adr/0003-tray-and-single-instance.md`に記録します。

### Q14. 設定画面の形

**不足している情報**：通知領域の小型パネルだけで全設定を行うか、独立した設定ウィンドウを持つかが決まっていません。

**決定**：2026年7月15日、MVPは通常のWPFウィンドウと通知領域メニューを組み合わせます。
起動時はウィンドウを表示し、閉じる操作は通知領域へ格納します。
プロセスを止める操作は画面または通知領域の「終了」へ限定します。
通知領域メニューには画面表示、手動スリープ防止、AC限定、再読み込み、終了を置きます。

**再判断**：2026年7月15日の実機確認後、通常ウィンドウを主画面とする判断を見直しました。
製品上の目標は、通知アイコンの近くへ表示する小型コントロールセンター型フライアウトとします。
通常ウィンドウは縮小して残すのではなく、標準タイトルバー、タスクバー表示、常設フッターを持たない開閉モデルへ置き換えます。
実装方式は`docs/adr/0004-tray-control-center-surface.md`をProposedとし、Windows 11の技術検証後に確定します。
レビューは`docs/reviews/2026-07-15-tray-control-center-ui.md`、実装計画は`docs/plans/2026-07-15-tray-control-center-redesign.md`に記録します。

### Q15. 自動起動、配布、更新

**不足している情報**：Startup Task、レジストリ、ショートカットの選択と、MSIXまたは別インストーラーの選択がありません。

**決定**：2026年7月18日、配布と自動更新にはVelopackを採用しました。
通常利用はWindows 11 x64向けの自己完結型Setupとし、GitHub Releasesの安定版だけを更新元にします。
起動後の確認で更新が見つかった場合はフライアウトへ通知し、利用者が承認した場合だけダウンロードして再起動します。
Portable ZIPは試用向けに残しますが、アプリ内更新は無効です。
コード署名と自動起動は引き続き未決です。
理由と制約は`docs/adr/0006-velopack-updates.md`に記録します。

### Q16. ログの既定値と保存期間

**不足している情報**：診断ログの保存有無、保存期間、利用者が削除するUIが決まっていません。

**推奨案**：エラーと状態遷移のメタデータだけをローカルへ7日間保存し、設定画面から開く、消去する操作を提供します。
通常ログに完全な作業パスとセッションIDを残しません。

**決定**：未決。

### Q17. 手動ディスプレイ消灯

**不足している情報**：即時消灯では誤操作を取り消せず、フライアウトを閉じた後も予約が残ると利用者の意図に反して消灯する可能性があります。

**決定**：2026年7月17日、3秒のキャンセル可能なカウントダウンを持つ「ディスプレイをオフ」をMVPへ追加します。
ボタンの再押下、Escape、外側クリック、フライアウトを閉じる操作でキャンセルし、ゼロへ到達した場合だけフライアウトの終了モーション後に一度実行します。
操作はスリープ防止と独立させ、電源プランや保存設定を変更しません。
Windows実装は`WM_SYSCOMMAND / SC_MONITORPOWER`をDisplayLight自身のウィンドウへ渡し、接続中のディスプレイへ一括消灯を要求します。
実際の消灯と入力による再点灯は実機確認を残します。
判断理由は`docs/adr/0005-manual-display-off.md`に記録します。
