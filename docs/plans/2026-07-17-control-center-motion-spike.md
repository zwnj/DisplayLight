# コントロールセンター型モーションの再検証

## 目的

Windows 11のコントロールセンターを録画した参照動画に合わせ、フライアウト全体がタスクバーの裏から現れる開閉モーションを検証します。
秒数の完全一致ではなく、完全非表示から始まること、面全体が同じ進捗で移動すること、開閉で異なる速度曲線を使うことを評価します。

## 完了条件

- 初回表示では、最終位置を先に描画せず、フライアウト全体がタスクバー側へ隠れた位置から開始する。
- 開くときは約250ミリ秒のcubic ease-outで最終位置へ移動する。
- 閉じるときは約170ミリ秒のcubic ease-inでタスクバー側へ隠れ、完了後に`Hide`する。
- 背景、外周、影、内容を含む`FlyoutSurface`全体が同じ座標で移動する。
- モーション中の不透明度は100%を維持する。
- 位置更新はWPFの描画フレームへ同期する。
- 面が最終位置へ到達した後、内容だけを約200ミリ秒でフェードインする。
- 内容の必要な高さが変わる場合、通知アイコン側の辺を基準に約180ミリ秒で伸縮する。
- サイズ変更中は内部操作を受け付けず、内容をウィンドウ境界でクリップする。
- `SystemParameters.ClientAreaAnimation`が無効な場合は即時に開閉する。
- 下、上、左、右タスクバーの隠れる方向と距離を純粋計算のテストで固定する。
- `pwsh -NoProfile -File ./scripts/verify.ps1`が成功する。

## 対象外

- Windows Shellの非公開APIは使用しない。
- UIの情報構造、電源操作、スライダーを変更しない。
- 参照動画のフレーム位置をピクセル単位で完全再現しない。
- 水平配置の通知アイコン中央揃え規則は変更しない。

## 実装方針

トップレベルウィンドウは最終位置へ一度だけ配置し、開閉中は位置を変更しません。
背景、外周、内容を含む`FlyoutSurface`を一つのVisualとして扱い、ウィンドウ境界によるクリップと`TranslateTransform`で表示範囲を変えます。
開始および終了オフセットは固定DIPではなく、タスクバーに垂直な方向のVisual寸法から算出します。

開く曲線は`1 - (1 - t)^3`、閉じる曲線は`t^3`とします。
参照動画で支配的なのはフェードではなくタスクバーによる遮蔽だったため、ウィンドウの不透明度は変化させません。
固定間隔の`DispatcherTimer`は使用せず、`CompositionTarget.Rendering`でVisualの変換を更新します。
外周面を描画する`FlyoutSurface`は常に不透明とし、内側の`FlyoutContent`だけを移動完了後にフェードします。
Desktop Acrylicはトップレベルウィンドウ全体へ描かれるため、Visualの移動中だけ無効化します。
Visualが最終位置へ到達した後にDesktop Acrylicを戻し、内容を約200ミリ秒でフェードします。
初回表示後はWPFの`SizeToContent`による瞬間的な高さ変更を止め、`FlyoutContentPanel`の必要サイズを監視します。
必要サイズが変わった場合は、通知アイコンを基準にした最終配置を再計算し、ウィンドウ位置と寸法を同じsmoothstep進捗で補間します。
開閉や内容フェードの途中に発生したサイズ変更は破棄せず、進行中のモーションが完了してから実行します。

## 2026年7月17日のサイズ変更背景修正

`SetWindowPos`でHWNDの高さが先に増え、WPFの`FlyoutSurface`が再描画されるまでDesktop Acrylicの灰色面が下部へ露出することを実機画像で確認しました。
サイズ変更中だけ`HwndSource.CompositionTarget.BackgroundColor`を`SurfaceBrush`と同じ不透明RGBへ切り替え、新しいクライアント領域を同色で初期化します。
最終サイズへ移動した後は次の描画フレームを待って透明へ戻し、通常時のDesktop Acrylicを維持します。

Windows 11 build 26100以降では`DWMWA_REDIRECTIONBITMAP_ALPHA`を利用できますが、System BackdropはHWND全体へ適用されます。
固定された大きい透明ホストへ単純に置き換えるだけでは未使用領域のBackdropを隠せないため、今回の修正では採用しません。
将来、DirectCompositionなどでBackdropを含む単一サーフェスを構成する場合は、トップレベルHWNDの連続リサイズ廃止を再検討します。

## 2026年7月17日の敵対的検証

利用者の録画を30fpsで分解した結果、開く途中の灰色帯は約100ミリ秒にわたり、ウィンドウの移動に合わせて下端から縮んでいました。
灰色帯は`#535353`、通常面は`#201F22`であり、単発の描画遅延ではなく別の合成背景面です。

`SetWindowPos`によるHWND移動とWPFのリダイレクト面は同じ描画トランザクションに含まれません。
クローク、`DwmFlush`、クリア色の変更はアニメーション開始前の面だけを調整するため、移動中の各フレームで生じる差を解消できません。

この結果を受け、開閉アニメーションからトップレベルHWNDの連続移動を廃止します。
固定したHWND内で単一Visualを移動し、DWM背景とWPF面が異なる座標で合成される経路をなくします。

## 2026年7月17日の伸縮動作修正

拡張途中では`ScrollViewer`が一時的なオーバーフローを検知し、右端へ縦スクロールバーを表示していました。
テンプレート内のスクロールバーを後から透明化する方式では表示を確実に防げなかったため、`VerticalScrollBarVisibility=Hidden`として常に描画しない方式へ変更しました。

初回実装は固定された外側ウィンドウのDesiredSizeを再計測していたため、折りたたみ後も展開時の高さが目標値として残る場合がありました。
内側コンテンツを高さ無制限で再計測し、その`DesiredSize`と外周境界線から目標高を直接算出するよう修正しました。
さらに展開状態と状態メッセージの変更通知からサイズ更新を明示的に要求し、拡張と縮小の両方向を同じ補間処理へ通します。
伸縮完了時は非表示`ScrollViewer`の内部オフセットを先頭へ戻し、最終サイズでレイアウトとマウス位置を再評価してから操作を再開します。
伸縮アニメーション内から次の再計測を開始すると、先行処理がキャンセルされて`IsHitTestVisible=false`だけが残る競合がありました。
再計測の開始は伸縮処理全体の終了後へ移し、成功、中断、例外のいずれでも`finally`から入力を復元します。

## 実機で確認すること

- トレイアイコンのクリックから、完成位置のフレームを挟まずに出現すること。
- タスクバーがフライアウトより前面に見え、下端を自然に隠すこと。
- 開く動きが急なジャンプや長い減速尾に見えないこと。
- 閉じる動きの完了前にウィンドウが消えないこと。
- 連打、外側クリック、Escapeで表示状態が破綻しないこと。
- 100%、150%、200% DPIと複数モニターで開始位置がタスクバー裏になること。
